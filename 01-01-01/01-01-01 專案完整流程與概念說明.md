# 01-01-01 專案完整流程與概念說明

## 1. 專案定位

`01-01-01` 是自 `01-01` 複製出的獨立工作副本，用於延伸策略研究、最佳化、驗證與決策流程的開發與實驗。此副本的主要目的，是在不回頭修改既有主線專案的前提下，逐步將原本以「單次最佳化 / 單次驗收」為主的工作台，升級為具備多層驗證觀念的研究與決策工作台。

本專案的修改範圍限定於：

- `C:\Users\User\Documents\mqquant\01-01-01`

本專案不應修改下列既有專案：

- `C:\Users\User\Documents\mqquant\xs-core-engine`
- `C:\Users\User\Documents\mqquant\01`
- `C:\Users\User\Documents\mqquant\02`

### 1-1. 目前定位

目前 `01-01-01` 的定位，是一個以單頁 Streamlit 介面為主的策略研究工作台，整合以下任務：

- 參數最佳化
- 開發區審查
- 單次 OOS 驗收
- 多輪 WFO 驗證
- 風險提示與輔助判讀
- 最新策略輸出與 artifact 回填
- Forward Test 前測紀錄

需要特別說明的是，`01-01-01` 已進入「五層驗證架構」的方向，但不同層級的成熟度並不相同。目前有些功能屬於正式可用層，有些屬於基本版、近似版、紀錄層或後續擴充方向，尚未形成完整的一鍵式全自動流水線。

### 1-2. 交易與回測核心前提

本專案目前依循以下交易與回測核心前提，這些前提屬於整個研究框架的最高優先規範：

1. 商品：大台指
2. 週期：1 分 K
3. 交易型態：日當沖，不留倉
4. 執行時點：當根 open 一出現，即用當根 open 進行判斷與執行
5. 判斷所依據的資料，只能來自前一根或更早、已完成且定錨的資料
6. 禁止使用當根尚未完成的 high / low / close / volume 來決定當根是否進出場
7. 回測基準滑價預設為 2 點
8. 壓力測試滑價為 3 / 4 / 5 點

白話例子如下：

例如 09:13 這根 1 分 K 的 open 一出現，就在 09:13 open 這個時點進行判斷與執行；但判斷時只能使用 09:12 或更早、已完成且定錨的資料。不能使用 09:13 這根尚未走完的 high / low / close / volume，回頭決定 09:13 open 是否應該進出場。換句話說，09:13 的進出場判斷必須建立在 09:12 以前已經確定的資訊上，而不是建立在 09:13 事後才知道的資訊上。

### 1-3. 專案核心目標

本專案的核心目標，不是單純找出總回測第一名，而是盡量淘汰樣本內漂亮、未來易失效的假冠軍，留下較穩健、較不容易死亡的候選策略。

- `app.py`

## 3. 專案核心概念

`01-01-01` 的研究邏輯，不是把最佳化跑完後，直接把單次回測第一名視為候選上線策略，而是建立一個逐層收斂的研究流程。其核心概念包括：

- 最佳化的作用，是產生候選，不是直接決策
- 單次 OOS 的作用，是初篩，不是直接升格
- 多輪 WFO 的作用，是檢查跨視窗穩定性
- gap 的作用，是降低訓練區與測試區邊界過近帶來的污染風險
- PBO / DSR 的作用，是提供過度最佳化風險提示
- Forward Test 的作用，是在正式升格前留下更貼近真實交易環境的觀察與決策紀錄

因此，本專案目前應被理解為：

- 一個以研究為主、決策為輔的工作台
- 一個逐步擴充中的五層驗證框架
- 一個不以單一最佳參數或單次漂亮結果作為唯一依據的策略研究環境

## 5. 左側欄位與設定意義

左側欄位目前承擔「設定研究條件」的角色，主要可分為以下區塊。

### 5-1. 快速流程

此區塊的目的是讓使用者知道操作順序，而不是做分析。當前主線操作順序為：

1. `開始執行`
2. `步驟 2：測試驗證區`
3. `步驟 3：計算多輪 WFO`
4. 查看 `多層策略驗證與決策系統`

### 5-2. 資料路徑

主要欄位：

- `XS 路徑`
- `M1 路徑`
- `D1 路徑`
- `參數範圍 preset`

作用如下：

- `XS 路徑`：指定目前策略程式
- `M1 路徑`：指定 1 分 K 資料
- `D1 路徑`：指定日線資料
- `參數範圍 preset`：指定本輪最佳化使用的參數範圍

目前系統已具備「最新策略與 preset 自動回填」機制。當最佳結果落盤後，系統會另寫出最新策略檔與 preset 檔，並自動更新左側路徑，降低人工切換成本。

### 5-3. 搜尋模式

此區塊用於指定最佳化的搜尋模式，例如智慧搜尋等。其作用是控制候選組合的產生方式與測試範圍。此區塊屬於 optimizer 的搜尋策略設定，而非驗證層設定。

### 5-4. 執行設定

主要欄位包括：

- 本金
- 單邊滑價
- 開發區年份
- 最大背景程序數
- CPU 使用率上限
- 記憶體使用率上限
- 保留前幾名
- 每個參數保留前幾名
- WFO 訓練窗(年)
- WFO 測試窗(年)
- WFO 滑動步長(年)
- WFO gap 天數

其中需要特別區分：

- `開發區年份`：控制單次最佳化與單次 OOS 所使用的大區段切分
- `WFO 設定`：只在使用者按下 `步驟 3：計算多輪 WFO` 時生效，不會影響單次最佳化按鈕本身

### 5-5. 硬性過濾

主要欄位包括：

- 最少交易筆數
- 最低總報酬
- MDD 門檻模式
- 固定 MDD 或 MDD 範圍測試

作用如下：

- 在候選組合中先做基礎淘汰
- 將明顯不合規或風險過高的組合排除
- 協助降低後續驗證與分析的噪音

### 5-6. 研究風險提醒

左側欄位雖然提供較高彈性的搜尋與過濾能力，但研究空間愈大，愈容易出現假冠軍。需注意以下風險：

- 搜尋空間越大，越容易出現樣本內特化結果
- 技術指標越多、參數排列組合越複雜，越容易出現資料探勘偏差
- 因此當搜尋空間擴大時，更需要依賴：
  - WFO
  - 穩定區分析
  - PBO / DSR 風險提示
  - Forward Test 前測
  來過濾假冠軍
- 系統不應以單一最佳參數或單一最優候選，作為唯一上線依據

## 6. 主流程：使用者實際操作順序

### 步驟 1：開始執行

使用者設定左側條件後，按下 `開始執行`，系統會：

1. 讀取目前路徑、參數與過濾條件
2. 建立背景 job request
3. 啟動背景 worker
4. 執行參數搜尋與最佳化
5. 持續更新目前最佳候選與進度

此階段的主要目標，是先從搜尋空間中產生可供審查的候選。

### 步驟 2：單次 OOS 驗收

當最佳化有結果後，使用者按下 `步驟 2：測試驗證區`。系統會：

1. 取目前最佳候選參數
2. 於 holdout 區段執行單次樣本外驗證
3. 產生：
   - 單次 OOS 報酬
   - 單次 OOS MDD
   - 單次 OOS 勝率
   - 單次 OOS 交易數
   - verdict
   - 滑價壓力測試結果

這一層的定位是初篩。其目的是先淘汰最明顯只會樣本內好看的候選，而不是直接決定是否上線。

### 步驟 3：多輪 WFO 驗證

當單次 OOS 完成後，使用者可按下 `步驟 3：計算多輪 WFO`。系統會：

1. 按 WFO 設定切出多個 train / test fold
2. 在每一輪 train 區段重新最佳化
3. 保留每輪 Top N 候選
4. 將候選丟到該輪 test 區段做 OOS 驗收
5. 記錄各輪 OOS 結果與通過情況

此步驟的目的，是檢查策略是否只適合單一市場區段，或能在多個時間窗中維持一定的穩定性。

### 步驟 4：查看多層策略驗證與決策系統

當前兩步資料具備後，使用者再查看 `多層策略驗證與決策系統`。此區塊會彙整：

- 單次 OOS
- 多輪 WFO
- gap 使用情況
- PBO / DSR 風險提示
- 滑價壓測
- 穩定區 / 尖峰警示
- Forward Test 前測入口或紀錄提示

需要注意的是，這個區塊目前已具備五層驗證框架，但尚未完全收斂為一鍵式自動決策 pipeline。

## 7. 研究期間與切窗概念

### 7-1. 單次研究期間

單次研究期間由 `resolve_0101_research_periods()` 解析，主要切分為：

- 開發區（development）
- 驗證區（holdout）

`開發區年份` 會影響 train 區段長度，而剩餘區段則作為單次 OOS 驗收使用。

### 7-2. 多輪 WFO 期間

WFO 切窗由 `build_0101_wfo_windows()` 負責，主要參數包括：

- 訓練窗長度
- 測試窗長度
- 滑動步長
- gap 天數

WFO 與單次 OOS 的差異在於：

- 單次 OOS：是一組較大的開發區 / 驗證區切分
- WFO：是在研究期間內切出多組較小的 train / test 視窗，重複驗證穩定性

## 8. 單次 OOS、多輪 WFO、五層驗證之關係

### 8-1. 單次 OOS 的定位

單次 OOS 是初篩層。其主要目的不是決定最終上線，而是先淘汰最明顯只會樣本內好看的候選。單次 OOS 不能單獨決定策略升格。

### 8-2. 多輪 WFO 的定位

多輪 WFO 是穩定性驗證層。其目的在於讓候選策略在多個 train / test 視窗中反覆接受驗證，比單次 OOS 更能看出策略是否只適合某一段市場。

### 8-3. 五層驗證的定位

五層驗證是逐層淘汰假冠軍的決策框架，而不是一開始就直接挑第一名。其目前的框架可理解為：

1. 單次 OOS 初篩
2. 多輪 WFO 穩定性驗證
3. gap 防污染
4. PBO / DSR 過度最佳化風險提示
5. Forward Test 前測

### 8-4. 目前成熟度說明

目前五層驗證已建立框架，但成熟度不一，應做如下理解：

- 第 1 層單次 OOS：屬於目前正式可用功能
- 第 2 層多輪 WFO：已具備基本可用的骨架與結果輸出
- 第 3 層 gap：已實作基本版；purge / embargo 尚未完整建立
- 第 4 層 PBO / DSR：目前為實務代理指標 / 近似評分，不是嚴格學術定義版
- 第 5 層 Forward Test：目前偏紀錄與決策輔助層，尚未形成完整生命週期引擎

因此，`01-01-01` 目前應被視為「已進入五層驗證架構」，而不是「五層驗證已完整正式落地」。

## 9. 目前已實作的風險控制與防過度最佳化概念

### 9-1. MDD 門檻與範圍測試

系統目前已支援：

- 固定 MDD 門檻
- MDD 範圍測試（sweep）

其作用是在同一批回測結果上，以不同 MDD 門檻分層觀察候選表現，而不需要每一個門檻重跑一次最佳化。

### 9-2. 穩定區與參數家族觀念

系統目前已開始納入：

- `plateau_score`
- 候選群的參數分布
- 參數家族表現比較

其目的在於降低單點最優的誘惑，避免把只在單一參數點特別亮眼的候選，誤判為真正穩健的策略。

### 9-3. 滑價壓力測試

單次 OOS 後，系統目前可檢視：

- 基準滑價：2 點
- 壓力滑價：3 / 4 / 5 點

並提供各滑價條件下的：

- 報酬
- MDD
- Sharpe
- 是否仍為正報酬

此機制的目的，是觀察策略在交易成本上升時是否快速失效。

### 9-4. PBO / DSR 風險提示

目前系統已有 PBO / DSR 的實務代理指標與近似評分，用於提示過度最佳化風險。需要明確說明：

- 目前的 PBO / DSR 為實務代理指標 / 近似評分
- 其目的在於提供過度最佳化風險提示
- 不可視為嚴格學術定義版
- 在策略最終升格時，PBO / DSR 只能作為輔助依據之一，不能單獨作為唯一決策依據

## 11. 目前畫面區塊的閱讀方式

目前畫面已開始朝「流程導向」調整，但仍保留部分研究工作台特性。建議閱讀順序如下。

### 11-1. 先看「現在要做什麼」

這是操作導引區塊，用來明確指出目前下一步該按哪個按鈕，例如：

- 先跑最佳化
- 再跑單次 OOS
- 再跑 WFO
- 最後再看決策系統

### 11-2. 再看「單次驗證總覽」

這裡是單次 OOS / 開發區守門的摘要區塊，定位是初篩與補充資訊，而不是最終決策區。

### 11-3. 再看「步驟 2：單次 OOS 驗收」

這裡是單次 OOS 的入口與結果來源區塊，用來完成初步的樣本外驗證。

### 11-4. 再看「步驟 3：多輪 WFO 驗證」

這裡是多輪 WFO 與 gap 設定真正發揮作用的區塊。

### 11-5. 之後再看「多層策略驗證與決策系統」

這裡才是目前最接近最終判斷的主區塊。PBO / DSR、滑價壓測、穩定區提示與 Forward Test 相關資訊，應在此區塊綜合解讀。

### 11-6. 補充分析與詳細頁籤

其餘長表與詳細頁籤，如：

- 圖表
- 策略組合
- KPI
- 交易明細
- 開發區審查
- 進階補充分析

主要定位是複核與查證，不是第一眼先看的資訊。

## 12. 背景任務與狀態流

本專案的最佳化主流程，並非完全在 Streamlit 前台同步執行，而是透過背景 worker 機制處理。其主要流程如下：

1. UI 收集左側設定
2. 建立 job request
3. 啟動背景 worker process
4. 背景 worker 呼叫 optimizer
5. 持續將進度、候選、摘要、錯誤與狀態寫回 job state
6. 前台 Streamlit 週期性讀取 job state
7. 工作完成後保存最佳結果與相關 artifact

目前已具備的穩定性控制包括：

- heartbeat
- stop flag
- stale job cleanup
- force stop
- 背景 worker 監控顯示

其目的是降低瀏覽器關閉後背景 Python worker 長時間殘留的風險。

## 14. 目前仍未完全成熟的能力

### 14-1. 五層驗證仍非完整一鍵式 pipeline

目前五層驗證已具備框架與部分功能，但仍不是「按一次就完整自動跑到最終建議」的一鍵式 pipeline。實際上，使用者仍需分段操作：

- 先跑最佳化
- 再做單次 OOS
- 再跑多輪 WFO
- 再綜合查看決策系統

### 14-2. gap 為基本版，purge / embargo 尚未完整建立

目前第 3 層以 gap 為主，已支援 train / test 中間留 gap。這能在一定程度上降低邊界貼近造成的污染風險。

但 purge / embargo 結構目前尚未完整建立，因此：

- 不能將第 3 層理解為完整的防污染機制已正式完成
- 現階段應更準確地理解為「已實作 gap 基本版，purge / embargo 仍屬後續方向」

### 14-3. PBO / DSR 為近似版

目前 PBO / DSR 並非嚴格學術定義版，而是用於實務研究流程中的代理指標與近似評分。它們的定位是：

- 提供過度最佳化風險提示
- 補強單次 OOS 與 WFO 之外的風險視角

但它們目前不應被視為最終決策的唯一依據，也不宜被表述為正式學術版落地。

### 14-4. Forward Test 目前偏紀錄層

目前已建立前測紀錄檔與欄位結構，也可由 UI 寫入 Challenger 前測紀錄。但整體而言，Forward Test 目前仍偏向：

- 紀錄層
- 決策輔助層
- 人工管理層

而非完整自動化生命週期引擎。自動升格、狀態流轉、前測週期管理等，仍屬後續強化項目。

### 14-5. UI 仍兼具研究工作台與決策工作台雙重性質

目前畫面已開始朝「流程導向」與「決策導向」重排，但仍保留大量研究工作台特徵，例如：

- 補充長表較多
- 視圖層次仍偏深
- 詳細圖表與複核資訊仍與主流程共存

因此，現階段 UI 仍是「研究工作台 + 決策工作台」混合型態，尚未完全收斂為純決策首頁。

### 14-6. 候選策略爆量時的治理機制仍可再加強

當搜尋空間擴大、候選爆量時，系統目前雖已開始引入風險提示，但治理機制仍可進一步加強，特別是：

- 搜尋空間限制
- 複雜度懲罰
- 穩定區優先
- 假冠軍淘汰機制

也就是說，當技術指標與參數組合數量急遽上升時，僅靠單次排名與少量輔助指標仍不足以穩定控風險，未來仍需進一步強化候選治理策略。

## 15. 正式結語

`01-01-01` 目前的定位，不是「自動找回測第一名的工具」，而是「逐層驗證、逐層淘汰假冠軍、最後協助策略升格與前測決策的研究工作台」。

本專案的核心目標，不是單純找出總回測第一名，而是盡量淘汰樣本內漂亮、未來易失效的假冠軍，留下較穩健、較不容易死亡的候選策略。
